# ADR-004: Migration path from Kong + jumper to gateway-core

- **Status**: Accepted
- **Date**: 2026-04-20
- **Deciders**: Stargate platform engineering, SRE, zone owners
- **Related**: ADR-001 (Kong-free), ADR-002 (CRD control plane), ADR-003 (zone/JWT mesh)

## Context

The Stargate fleet runs **Kong + jumper** in every zone today. Dozens of zones, hundreds
of routes per zone, thousands of consumers, production traffic at each zone. A big-bang
cutover is not feasible. We need a plan that:

- Lets zones migrate independently, at their own pace.
- Preserves wire compatibility between migrated and unmigrated zones throughout the
  migration (ADR-003 is what makes this possible).
- Converts existing Kong declarative config + jumper `application.yml` into CRDs
  automatically, without operator-authored YAML rewrites.
- Allows instant rollback per zone if something regresses.
- Finishes on a realistic timeline.

## Decision

We migrate **one zone at a time**, **one pod at a time within each zone**, using a
shadow-to-canary-to-full cutover. A dedicated migration tool converts existing config;
feature flags in the data plane let us toggle individual scenarios off if they misbehave.
Rollback is a single Helm value per zone.

### Tooling: `core-migration-tool`

A standalone Maven module that reads:

1. The zone's Kong declarative YAML
   (`/etc/kong/kong.yaml` equivalent or the Admin API dump).
2. The zone's `jumper/application.yml`
   (`/Users/A85894249/claude-code/gateway-jumper/src/main/resources/application.yml`
   is the schema reference).
3. The zone's Helm values (to learn Redis address, realm name, etc.).

...and emits:

- `GatewayZone.yaml` (one per zone, keyed from realm + issuer).
- `GatewayMeshPeer.yaml` (one per zone we mesh-to, from jumper's zone-to-issuer mapping).
- `GatewayRoute.yaml` files (one per Kong route, with the matching Kong service's
  `url` as `backendRefs`).
- `GatewayConsumer.yaml` + `GatewayCredential.yaml` files (from Kong consumers and their
  `jwt_secrets` / `key-auth-credentials`).
- `GatewayPolicy.yaml` files (from Kong plugins: `rate-limiting`, `cors`, `key-auth`,
  `jwt`, `request-size-limiting`, etc.).

The tool is deterministic and idempotent: running it twice on the same input produces
identical output. It emits a `migration-report.md` listing anything it couldn't convert,
with line-level pointers back to the source YAML. Unsupported plugin instances produce
explicit `TODO` markers rather than silent data loss.

Invocation:

```
core-migration-tool \
  --kong kong.yaml \
  --jumper jumper-application.yml \
  --helm values-zone.yaml \
  --output manifests/zone-aws/
```

### Rollout phases

Each zone goes through the same five phases. The whole fleet does phase N before any
zone moves to phase N+1.

#### Phase 0 -- Preparation (2 weeks)

- Deploy `gateway-core` controller into one lab zone alongside Kong + jumper (no traffic).
- Run `core-migration-tool` against a real zone's config; review output with zone owners.
- Fix tool gaps until a "happy path" zone converts with zero `TODO` markers.
- Land `core-testkit` replay fixtures captured from production Kong traffic.

#### Phase 1 -- Shadow traffic (per zone, ~1 week)

- Deploy `gateway-core` data-plane pods **behind** the Kong+jumper pods, receiving
  **mirrored** traffic only (via Kong's `request-termination` or an external
  traffic-mirror). Responses are discarded.
- Apply the CRDs generated by `core-migration-tool`.
- Compare responses (status, headers, body hash) via a diffing sidecar. Investigate every
  divergence.
- Success criterion: <0.1% divergence for 48h of business-hours traffic, zero diverges
  on auth and mesh paths.

#### Phase 2 -- Canary cutover (per zone, ~1 week)

- Switch **one pod** per zone (out of N) to serve real traffic from `gateway-core`
  instead of Kong + jumper. Use the existing service mesh / Helm weight to route ~5% of
  traffic there.
- Watch error rate, p99 latency, mesh-token cache hit rate, zone-health pub/sub
  connectivity, the `zone.health.status` gauge
  (`/Users/A85894249/claude-code/gateway-jumper/src/main/java/jumper/service/ZoneHealthCheckService.java:43-56`).
- Rollback trigger: **any** of: error rate +0.5% over baseline, p99 +10 ms over baseline,
  failed mesh-token fetch > 0.1%.

#### Phase 3 -- Majority cutover (per zone, ~1 week)

- Ramp the canary from 5% -> 25% -> 50% -> 90% over four business days.
- Keep at least one Kong+jumper pod warm for instant rollback.
- Every ramp step has a 2h soak before the next.

#### Phase 4 -- Decommission (per zone, ~1 week)

- 100% traffic on `gateway-core`.
- Scale Kong + jumper pods to zero (but keep the Deployments for 30 days).
- Remove Kong admin API consumers; revoke Kong admin credentials.
- Close the zone's migration ticket.

#### Phase 5 -- Fleet cleanup (global, post last zone, ~2 weeks)

- Archive the jumper repository to read-only mode.
- Remove `jumper_config` header parsing from `gateway-core` (ADR-003 carries it forward
  as a compatibility surface during migration).
- Remove Kong artifacts from the Helm chart. Publish `gateway-core` as the default.

### Timeline

Assuming ~12 production zones and 2 parallel tracks of zone owners:

| Phase                         | Duration            |
|-------------------------------|---------------------|
| 0 -- Preparation              | 2 weeks (one-time)  |
| 1 -- Shadow, per zone         | 1 week              |
| 2 -- Canary, per zone         | 1 week              |
| 3 -- Majority, per zone       | 1 week              |
| 4 -- Decommission, per zone   | 1 week              |
| 5 -- Fleet cleanup            | 2 weeks (one-time)  |

With 2 concurrent zones in flight, end-to-end fleet migration lands in roughly **16--18
weeks** of calendar time, plus the bootstrap and cleanup weeks. The controller is
provisioned globally in phase 0; zones only move their data plane.

### Rollback

Rollback has three granularities, each trivial to execute:

1. **Per-pod** (phase 2 and 3). Helm weight back to 0 for the `gateway-core` pod; Kong +
   jumper picks up the traffic. Resolution time: seconds.
2. **Per-zone** (phase 2, 3, 4). Helm value `gatewayCore.enabled=false`. The zone returns
   fully to Kong + jumper. Resolution time: one rolling restart.
3. **Per-scenario** (any phase). Runtime feature flags in the data plane can disable
   individual scenarios (mesh, external OAuth, basic auth, Spectre) and fall back to
   proxying raw. Used for emergency mitigation when a specific token flow misbehaves.

**Wire compatibility (ADR-003) is what makes per-zone rollback safe**: a rolled-back zone
continues to interoperate with its neighbours regardless of which stack they run.

### Risks and mitigations

| Risk                                                               | Mitigation                                                                  |
|--------------------------------------------------------------------|-----------------------------------------------------------------------------|
| Undetected behavioural divergence from Kong                        | 48h shadow-traffic diff, automated replay via `core-testkit`.               |
| Kong plugin we didn't anticipate                                   | `core-migration-tool` fails loudly on unknown plugins; zone owner signs off before phase 1. |
| Rate-limit implementation behaves differently under burst          | Shadow traffic measures burst response; `GatewayPolicy.rateLimits` has a "compat-kong" mode for phase 1/2. |
| Mesh incompatibility discovered mid-rollout                        | ADR-003 wire compatibility plus per-zone rollback. We also run an end-to-end mesh-interop test in CI using a jumper container as one peer. |
| Controller outage during rollout                                   | Controller runs HA; last-known-good snapshot is durable on data-plane pods. |
| Operator accidentally commits an invalid CRD                       | Admission webhook (ADR-002) rejects at the API boundary.                    |
| Hidden coupling to Kong `x-consumer-*` headers downstream          | `core-inbound-auth` emits the same header set; shadow-traffic diffing catches any drift. |

## Consequences

### Positive

- **Safe, boring migration.** No flag day; each zone moves when its owners are ready.
- **Instant rollback per zone.** Mitigates the "we only discover it in production"
  failure mode.
- **Automated config conversion.** Operators do not hand-translate Kong YAML to CRDs.
- **Wire compatibility keeps the mesh stable** across migrated and unmigrated zones
  (ADR-003).

### Negative

- **Timeline is real.** 4-5 months of calendar time with two parallel tracks. The team
  cannot treat `gateway-core` as "done" until the last zone is migrated.
- **Two stacks to support in parallel** until phase 5 completes. Cross-cutting bug fixes
  may need to land in both for a period.
- **Migration tool is itself code to maintain.** We retire it after phase 5 along with
  the `jumper_config`-header compatibility path.

### Neutral

- **Observability is doubled briefly.** Kong metrics and `gateway-core` metrics both
  flow into the same Prometheus during overlap phases. We add a `stack` label to
  disambiguate.

## Success criteria

We declare the migration complete when **all** hold:

- All production zones run `gateway-core` exclusively for at least 14 days.
- The Kong admin API is not reachable from any production cluster.
- The jumper repository is archived.
- The `core-migration-tool` module is moved to `contrib/` and marked end-of-life.
- SLOs for p99 latency, error rate, and mesh-token cache hit rate are met or improved
  versus the pre-migration baseline for 30 consecutive days.
